## Problem Statement

Existing chess analysis tools (Lichess, Chess.com, Chessigma) all rely on Stockfish to
evaluate positions. They tell you WHAT move is best, not WHY. A player uploading their
game gets "Nf6 was better (+0.5)" — no connection to patterns, grandmaster precedent,
or opening theory. The result is engine output a human can't learn from.

chess_rag fills that gap: upload a PGN, pick a move, get AI commentary that cites real
similar positions from chess history, names the pattern you fell into, and explains the
idea — grounded in a real knowledge base of annotated games and opening theory.

## What Makes This Cool

- "Magnus played this exact structure in 2021 and won by exploiting the e5 weakness"
  is not something Stockfish can say. This is.
- Zero existing chess RAG projects in the public space — this is genuinely novel.
- The knowledge base is public, free, and rich: ECO opening codes, Wikipedia chess
  articles, annotated PGNs from Lichess Studies, master game collections.
- FAISS vector index committed to the repo means zero infrastructure cost.
- Hosted free on Hugging Face Spaces with a permanent shareable URL.

## Constraints

- $0 hosting cost, $0 API cost
- Python stack (LangChain/LlamaIndex familiarity)
- No OpenAI key required — open source LLMs (Ollama locally, Groq free tier hosted)
- Embeddings via sentence-transformers (local, no key needed)
- Knowledge base: publicly available chess data only (no copyrighted books)
- Must be publishable and usable by others

## Premises

1. Goal is a portfolio/resume project that demonstrates RAG architecture end-to-end.
   Feature richness is secondary to architectural clarity.
2. Hosting cost must be $0. System uses HF Spaces + FAISS (file-based vector index).
3. Knowledge base is public chess data only.
4. No API key required — open source LLMs only. Ollama for local, Groq free tier for hosted.
5. The value is context (cited positions, patterns, opening theory), not better engine eval.

## Approaches Considered

### Approach A: CLI + Local ChromaDB
Minimal pipeline, no UI, local only. Fast to build, cleanest architecture,
no visual demo. Good for learning but weak resume artifact.

### Approach B: Streamlit + FAISS + HF Spaces (CHOSEN)
Web app with file upload, move picker, AI analysis panel. Hosted free with
permanent URL. Pre-indexed FAISS store committed to the repo.

### Approach C: Full Agentic RAG
LangChain agents with named tools. More complex, less predictable for demos.
Elements absorbed into B as structured retrieval functions.

## Recommended Approach

**Streamlit web app on Hugging Face Spaces, FAISS vector store, structured retrieval
functions named like agent tools.**

The retrieval layer is implemented as named Python functions:
- `retrieve_opening_theory(fen: str) -> list[str]` — finds matching ECO codes + theory
- `retrieve_similar_positions(fen: str, n: int) -> list[Game]` — finds similar annotated games
- `retrieve_pattern_explanation(pattern_name: str) -> str` — fetches pattern docs

These functions are called explicitly (not by an agent) but are named and structured so
you can explain the agentic architecture pattern in interviews. Easy upgrade path to full
LangChain agents later.

### Knowledge Base Sources
```
1. ECO Opening Database — 500 opening codes with descriptions (public domain)
   Source: https://github.com/lichess-org/eco
2. Wikipedia chess articles — openings, endgames, tactics (via Wikipedia API)
   ~200 articles on major openings, endgame theory, tactical patterns
3. Annotated PGN games — Lichess Studies (Creative Commons)
   Select ~1000 master games with human annotations from Lichess
4. Chess pattern descriptions — manually written or from chess wiki
   25-50 pattern descriptions (Greek Gift, Windmill, Back Rank, etc.)
```

### Ingestion Pipeline
```
rag/ingestion.py:
  1. Load source documents (PGN, markdown, text)
     - ECO database from lichess-org/eco (JSON → text chunks)
     - Wikipedia articles via Wikipedia API
     - Annotated PGNs from rag/fetch_lichess_games.py (see below)
     - Pattern descriptions from data/sources/patterns.md
  2. Chunk: games by move sequence + annotation, articles by paragraph
  3. Embed with sentence-transformers (all-MiniLM-L6-v2) — no API key
  4. Store in FAISS index → save to data/chess_index.faiss + chess_index_docs.pkl
  Run once at setup. Track index files via Git LFS locally. For HF Spaces deployment,
  use the Hugging Face Hub dataset approach: upload index files as a separate HF Dataset
  (free), download at app startup via `huggingface_hub.hf_hub_download()`.
  HF Spaces does not support Git LFS directly — Hub datasets are the correct path.
  Note: all-MiniLM-L6-v2 is a solid general-purpose baseline. If retrieval quality
        is poor in testing (wrong openings surfaced), the fix is better chunking
        strategy, not a different embedding model.

rag/fetch_lichess_games.py:
  - Uses Lichess public database API (no auth required)
  - Fetches games from Lichess Studies tagged with annotations
  - Filters: rated games, at least 80% moves with comments, master-level players
  - Outputs ~1000 annotated PGNs to data/sources/lichess_games/
  - Rate limit: 1 req/sec with exponential backoff on 429/5xx errors
    (3 retries, max 30s wait) to respect Lichess API guidelines
```

### Query Pipeline
```
app.py (Streamlit):
  1. User uploads PGN → parse with python-chess
  2. User selects a move from dropdown → extract FEN at that position
  3. Retrieval (rag/retriever.py):
     a. retrieve_opening_theory(fen) → top 3 matching ECO lines
     b. retrieve_similar_positions(fen) → top 3 annotated games with similar structure
     c. Pattern flow (two steps):
        i.  Ask LLM: "Given these retrieved games and this FEN, name the main
            tactical/strategic pattern." → returns e.g. "Greek Gift sacrifice"
        ii. retrieve_pattern_explanation("Greek Gift sacrifice") → 1 pattern doc
  4. Build augmented prompt via rag/prompts.py:
     - System prompt instructs: "Always cite the game, player, and year when referencing
       a historical position. Use the retrieved context as your primary source of truth."
     - Retrieved context is injected before the user's question
  5. Call LLM: Groq (if GROQ_API_KEY set) else Ollama local → stream to Streamlit
     - Groq quota exceeded: show friendly message "Daily limit reached. Run locally
       with Ollama: `ollama pull llama3.1 && streamlit run app.py`"
  6. Display: AI commentary + source citations (game name, player, year)
  Note: No visual chessboard in MVP. Move list in algebraic notation. Visual board
        deferred to v2 if users request it.
```

### patterns.md Format
Each pattern entry must follow this structure:
```markdown
## [Pattern Name]

[2-3 sentence overview: what it is and when it arises]

Key position indicators: [pawn structure, king safety, piece placement, open files]

After [key move/idea]: [what happens, what the attacker/defender gains]

Strategic idea: [one sentence on the underlying concept]

Example: [1-2 move sequence or FEN context]
```
